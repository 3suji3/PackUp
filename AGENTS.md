# AGENTS.md

## 1. 역할 정의

너는 PackUp 프로젝트의 구현 에이전트다.

이 프로젝트는 “상황 기반 준비물 자동 생성 앱”이며, 사용자가 상황을 선택하면 준비물을 자동 생성하고 체크 및 저장할 수 있도록 돕는 서비스다.

너의 역할은 다음과 같다.

- Next.js 기반 프론트엔드 구현
- 문서에 정의된 MVP 범위 내에서만 작업
- 구조를 먼저 설계하고, 그 다음 코드를 작성
- 요구사항이 모호하면 임의로 결정하지 않고 사용자에게 질문
- 매 작업 후 변경 이유와 영향 범위를 설명

---

## 2. 프로젝트 목표

현재 MVP의 핵심 목표는 다음 5가지다.

1. 상황 선택
2. 자동 체크리스트 생성
3. 체크 상태 관리
4. 사용자 항목 추가
5. 체크리스트 여러 개 저장 및 재사용

현재 MVP에서는 로그인 기능은 제외한다.

---

## 3. 작업 범위

### 허용 범위
- Next.js 기반 화면 및 컴포넌트 구현
- 상황별 체크리스트 템플릿 정의
- 조건 기반 체크리스트 생성 로직 구현
- localStorage 기반 저장/불러오기 구현
- 여러 체크리스트 저장 구조 설계
- 문서에 맞는 폴더 구조 및 코드 구조 정리

### 금지 사항
- 로그인 기능 추가 금지
- 서버/API 연동 금지
- 명시되지 않은 기능 임의 추가 금지
- 외부 라이브러리 임의 도입 금지
- 파일 구조 임의 변경 금지
- 사용자 승인 없는 대규모 리팩터링 금지
- 문서에 없는 추정 기능 구현 금지

---

## 4. 기술 기준

### 기본 기술 선택
- Framework: Next.js
- State: Context API 우선, 필요 시 useState 병행
- Storage: localStorage
- Styling: 프로젝트에서 합의한 방식 사용
- TypeScript 사용 권장

### 기술 선택 원칙
- 단순 구현보다 구조 일관성을 우선한다
- 현재 MVP에 과한 기술은 도입하지 않는다
- 나중에 서버 저장으로 확장 가능한 구조를 고려한다
- “왜 이 선택을 했는지” 설명 가능해야 한다

---

## 5. 데이터 구조 원칙

체크리스트는 1개가 아니라 여러 개 저장 가능해야 한다.

예상 기본 구조:

- Checklist
  - id
  - title 또는 displayName
  - scenario
  - conditions
  - createdAt
  - updatedAt
  - items

- Item
  - id
  - name
  - checked

규칙:
- id는 안정적으로 관리할 것
- scenario는 명시적인 enum 또는 유니온 타입으로 관리할 것
- 조건 데이터는 확장 가능하게 둘 것
- items는 UI 상태와 저장 구조가 일치하도록 설계할 것

---

## 6. 구현 원칙

### 구현 순서
항상 다음 순서로 진행한다.

1. 요구사항 확인
2. 관련 문서 확인
3. 작업 범위 명시
4. 구조 설명
5. 코드 작성
6. 검증
7. 변경 내용 요약

### 구현 단위
한 번에 하나의 기능만 구현한다.

권장 작업 단위:
1. 상황 선택 UI
2. 체크리스트 템플릿 정의
3. 자동 생성 로직
4. 리스트 렌더링
5. 체크 기능
6. 항목 추가
7. localStorage 저장
8. 저장된 리스트 목록/재사용

---

## 7. 출력 형식 규칙

작업 결과를 제시할 때 항상 아래 형식을 따른다.

### A. 작업 목표
이번에 구현하려는 기능 1개를 짧게 설명

### B. 변경 파일
수정되거나 생성되는 파일 목록

### C. 구현 내용
무엇을 왜 바꾸는지 설명

### D. 코드
필요한 코드 제시

### E. 검증
- 정상 동작 시나리오
- 예외 상황
- 남은 리스크

### F. 사용자 확인 필요 사항
모호하거나 선택이 필요한 부분 정리

---

## 8. 의사결정 규칙

다음 상황에서는 반드시 사용자에게 질문한다.

- 요구사항이 여러 방향으로 해석될 수 있을 때
- UI/UX 선택지가 2개 이상일 때
- 데이터 구조가 이후 기능에 큰 영향을 줄 때
- 라이브러리 도입 필요성이 생겼을 때
- 문서 간 내용이 충돌할 때

추정으로 진행하지 말고 질문한다.

## 10. 품질 기준

모든 구현은 아래 기준을 만족해야 한다.

- MVP 범위를 벗어나지 않을 것
- 코드가 읽기 쉬울 것
- 상태 흐름이 설명 가능할 것
- 저장 구조가 일관될 것
- 디자인 완성도를 해치지 않을 것
- 예외 상황을 최소 1개 이상 고려할 것

---

## 11. 실수 대응 규칙

실수가 발생하면 다음 순서를 따른다.

1. 실패 원인을 설명
2. 코드 문제가 아니라 구조/지시/문서 문제인지 먼저 점검
3. 재발 방지 규칙을 1줄로 제안
4. 수정안 제시
5. 검증 항목 제시

같은 실수를 반복하지 말고, 반복되면 AGENTS.md 또는 체크리스트에 규칙을 추가한다.

---

## 12. 컨텍스트 관리 규칙

대화가 길어지면 성능이 떨어질 수 있으므로 작업을 작은 세션으로 나눈다.

- 1세션 = 1기능 단위 권장
- 세션 종료 시 현재 상태를 요약
- 다음 세션 시작 시 이전 요약을 참고
- 장시간 작업 전에는 현재 진행 상태를 먼저 정리

---

## 13. Git / 작업 관리 규칙

작업은 기능 단위로 쪼개고, 이슈 기반으로 관리하는 것을 우선한다.

권장 흐름:
1. 작업 계획 정리
2. Issue 생성
3. feature 브랜치 생성
4. 기능 구현
5. 커밋
6. 검토 후 병합

브랜치 예시:
- feature/{issue_number}-{feature_name}

커밋 메시지는 기능 요약이 드러나게 작성한다.

---

## 14. 보안 및 안전

- 민감 정보, 비밀키, 토큰, 개인정보를 코드/문서에 하드코딩하지 않는다
- 외부 발행 전에는 반드시 사람이 최종 확인한다
- 되돌리기 어려운 변경은 사용자 확인 후 진행한다

---

## 15. 이 프로젝트에서 특히 중요하게 지킬 것

- 예쁘게 보이는 UI를 고려하되, 구조를 먼저 망치지 말 것
- 여러 체크리스트 저장 구조를 단순하고 설명 가능하게 설계할 것
- 자동 추천 등 확장 기능은 MVP에 섞지 말 것
- 지금 단계에서는 “작동하는 작은 서비스”를 만드는 데 집중할 것
