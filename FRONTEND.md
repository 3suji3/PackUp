# FRONTEND.md

## 1. 목적

이 문서는 PackUp 프론트엔드 구현 원칙과 현재 MVP 화면 구조를 정리한다.

목표:
- MVP 범위 안에서 단순하고 설명 가능한 구조를 유지한다.
- 화면 책임과 상태 책임을 느슨하게 분리한다.
- 다음 단계 확장 전까지 과한 상태 관리 도입을 피한다.

---

## 2. 폴더 구조

```text
src/
├─ app/
├─ features/
├─ types/
```

규칙:
- 공통 타입은 `src/types`에 둔다.
- 체크리스트 관련 생성 로직과 화면 컴포넌트는 `src/features/checklists` 아래에 둔다.
- 페이지는 화면 조합과 상태 연결을 담당한다.

---

## 3. 체크리스트 화면 구조

현재 체크리스트 화면은 다음처럼 나눈다.

- `src/app/page.tsx`
  - `currentChecklist` 상태를 소유한다.
  - 시나리오 선택, 체크 토글, 사용자 항목 추가 핸들러를 가진다.
- `src/features/checklists/components/ChecklistView.tsx`
  - 체크리스트 화면 표시를 담당한다.
  - 사용자 항목 입력창과 추가 버튼 UI를 가진다.
  - 입력값은 로컬 상태로 관리한다.

선택 이유:
- `page.tsx` 비대화를 막되, 아직 Context나 별도 store 없이도 흐름을 설명할 수 있다.
- 체크리스트 화면이 확장되더라도 feature 경로 안에서 UI를 계속 키울 수 있다.

---

## 4. 상태 관리 원칙

- 현재 단계에서는 `useState`만 사용한다.
- Context API와 localStorage는 아직 연결하지 않는다.
- 실제 체크리스트 데이터 변경은 `page.tsx`에서 처리한다.
- 화면에만 필요한 임시 입력값은 feature 컴포넌트 로컬 상태로 둔다.

---

## 5. 사용자 항목 추가 규칙

- 입력창 1개, 추가 버튼 1개만 제공한다.
- 빈 문자열 또는 공백만 있는 입력은 추가하지 않는다.
- 추가된 항목은 `currentChecklist.items` 끝에 붙인다.
- 새 항목은 `id`를 새로 만들고 `checked: false`로 시작한다.
- 항목 추가 시 `updatedAt`을 함께 갱신한다.

---

## 6. 금지 사항

- 로그인 기능 추가 금지
