# ARCHITECTURE.md

## 1. 개요

PackUp은 “상황 기반 준비물 자동 생성 앱”으로,  
사용자가 특정 상황을 선택하면 해당 상황에 맞는 체크리스트를 자동으로 생성하고 관리할 수 있는 서비스이다.

이 문서는 시스템의 구조, 기술 선택 이유, 데이터 흐름, 그리고 주요 설계 의사결정을 기록한다.

---

## 2. 시스템 구조 개요

### 전체 구조

Client 중심의 단일 애플리케이션 구조

- Next.js 기반 프론트엔드
- localStorage 기반 데이터 저장
- 서버 없는 구조 (MVP)

---

### 구조 특징

- 상태 중심 UI 구조
- 클라이언트에서 모든 로직 처리
- 저장/불러오기까지 프론트에서 담당

---

## 3. 아키텍처 선택 이유 (Why)

### 선택: Client-only 구조

#### 이유

- MVP 단계에서 빠른 개발 가능
- 서버 없이도 핵심 기능 구현 가능
- 배포 및 유지 비용 최소화

#### Trade-off

| 장점 | 단점 |
|------|------|
| 구현 속도 빠름 | 기기 간 데이터 공유 불가 |
| 구조 단순 | 데이터 영속성 제한 |
| 비용 없음 | 확장 시 서버 필요 |

---

### 선택: Next.js

#### 이유

- 페이지 기반 구조로 UI 설계 용이
- React 생태계 활용 가능
- 향후 서버 기능 확장 가능

#### Trade-off

| 장점 | 단점 |
|------|------|
| 구조 정리 쉬움 | 초기 학습 필요 |
| 확장성 좋음 | 설정 복잡도 존재 |

---

### 선택: localStorage

#### 이유

- 로그인 없이 상태 유지 가능
- 구현 간단
- MVP에 적합

#### Trade-off

| 장점 | 단점 |
|------|------|
| 빠른 구현 | 보안 취약 |
| 서버 필요 없음 | 데이터 동기화 불가 |
| 사용 간편 | 용량 제한 |

---

## 4. 데이터 구조

### 핵심 구조

```ts
Checklist {
    id: string
    scenario: "travel" | "school" | "gym"
    items: Item[]
    createdAt: number
}

Item {
    id: string
    name: string
    checked: boolean
}
````

---

### 설계 이유

* 여러 체크리스트 저장을 고려한 구조
* 상태와 저장 구조 일치
* 확장 가능성 확보

---

### 확장 고려

* 조건 (conditions) 필드 추가 가능
* 사용자 맞춤 템플릿 확장 가능
* 추천 시스템 연결 가능

---

## 5. 상태 관리 구조

### 선택: Context API + useState

#### 이유

* 전역 상태 공유 필요 (체크리스트 목록)
* Redux 대비 가벼움
* 학습 및 유지보수 용이

---

### 상태 구성

* checklistList (전체 리스트)
* currentChecklist (현재 선택된 리스트)

---

### Trade-off

| 장점    | 단점         |
| ----- | ---------- |
| 단순함   | 대규모 상태에 약함 |
| 빠른 구현 | 성능 최적화 제한  |

---

## 6. 데이터 흐름

### 기본 흐름

1. 사용자 → 상황 선택
2. → 체크리스트 생성 로직 실행
3. → 상태 저장
4. → UI 렌더링
5. → 체크 상태 변경
6. → localStorage 저장

---

### 저장 흐름

* 상태 변경 → localStorage 동기화
* 앱 시작 시 → localStorage에서 불러오기

---

## 7. 체크리스트 생성 로직

### 기본 구조

* scenario 기반 템플릿 매핑
* 조건 기반 항목 추가 (확장 예정)

---

### 예시

* travel → 기본 여행 리스트
* gym → 운동 리스트

---

### 설계 이유

* 단순 템플릿 기반 → 안정성 확보
* 조건 로직은 이후 확장 가능

---

## 8. UI 구조

### 페이지 구조

* / → 상황 선택
* /setup → 조건 선택
* /checklist → 리스트 화면
* /saved → 저장 리스트

---

### 설계 이유
